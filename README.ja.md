# Axiom UI

[English](./README.md) | 日本語

> An open-source UI library designed for the LLM as its primary consumer.
> LLMを第一の消費者として設計する、オープンソースUIライブラリ。

**Status: pre-v0 — 設計フェーズ。** 思想は確定済み。コンポーネントはまだ書かれていない。

---

## 反転（The Inversion）

既存のUIライブラリはすべて **人間のDX** に最適化されている。Axiom UI は、第一原理から **LLMによる生成・理解・改変** に最適化する。人間はその副次的な受益者になる。

`shadcn/ui` が「結果的に」AIフレンドリーなら、Axiom は **設計思想からAIファースト** である。

全ての設計判断は、次の一つの問いを通過しなければならない：

> 「この決定は、LLMがこのコンポーネントを **一度で正しく生成** する確率を上げるか、下げるか？」

上げないものは実装しない。

完全な[設計思想](./DESIGN_PHILOSOPHY.md)を読む。

## 設計原則

5段階のアルゴリズム — この順序で、不可逆に適用する：

1. **要求を疑う。** 全てのprop、コンポーネント、オプションは存在理由を説明できなければならない。
2. **削除する。** 最高のpropは「propがない」こと。1つのことをやる方法は1つだけ — バリアントはハルシネーションの燃料である。最高のコンポーネントは「コンポーネントがない」こと。
3. **簡素化する。** 削除を生き残ったものだけを磨く。
4. **高速化する。** コンポーネントあたりのtoken数とバイト数を計測し、CIでゲート化する。太ったコンポーネントはマージしない。
5. **自動化する — 最後に。** MCPサーバー、CLI、ドキュメント生成は、削除と一本化の後にのみ作る。

## アーキテクチャ

| 決定 | 帰結 |
|---|---|
| **1コンポーネント = 1ファイル** | 実装・型・ドキュメント・使用例・manifestを同一ファイルに co-locate。単一のcontext窓に収まる — 偶然ではなく、設計制約として。 |
| **型が契約** | `strict: true`、discriminated union、`any` 禁止。型が一次ドキュメントである。 |
| **コピー&オーン** | npmランタイム依存なし。CLIがコードをあなたのリポジトリに注入する。あなたがコードを所有し、LLMが自由に改変できる。ブラックボックスなし。 |
| **MCPファースト配布** | MCPサーバーをライブラリと同時に提供。エージェント（Claude Code、Cursor 等）がバージョン整合の取れた仕様をtoken無駄なく取得できる。 |
| **機械可読 manifest** | 各コンポーネントは `{ intent, props, states, a11y, examples }` を構造化データで持つ。エージェントがコンポーネントを「説明される」のではなく「選択して描画」できる — generative UI の土台。 |
| **ゼロコンフィグ・デフォルト** | propなしで正しく動く。エスケープハッチは用意するが、必須にはしない。 |

## 技術スタック（v0）

| 選択 | 理由 |
|---|---|
| **React** 専用 | 対象を絞り深く行く。抽象化レイヤを1つ削除する。 |
| **Tailwind CSS** | クラス文字列がそのままLLMに可読で、生成しやすい。 |
| **TypeScript strict** | 型 = 契約 = 一次ドキュメント。 |
| **CLI注入 + MCPサーバー** | ブラックボックス依存ゼロ。 |
| **ランタイム依存を最小化** | idiot index（本質コストに対する実コストの比）を低く保つ。 |

これらはv0の決定であり、思想の一部ではない。より良い理由があれば差し替えられる — ただし「選択肢を増やす」方向には決して変えない。

## Non-Goals（作らないもの）

- テーマの無限カスタマイズ機構（1つの良いデフォルトを持つ）
- 複数フレームワーク同時サポート（v0では）
- 「あらゆるユースケースに対応」を謳うことすべて
- 同じ目的を果たす2つ目のAPI
- 使われるか不明な機能のための設定オプション

## 成功の測定基準

主観ではなく数値で判定する：

- **token/component** — 平均コンポーネントが単一のcontext窓に収まること
- **bundle/component** — コンポーネント単位のバイト数を追跡し、退行させない
- **one-shot generation rate** — 主要KPI：LLMが修正なしで正しいコードを生成する割合
- **削除比率** — リリースごとに、追加した行数に対する削除した行数の比を記録する

## コントリビューション

全PRは、レビュー依頼の前に次に答えること：

1. これは何かを **削除** しているか？ もしくは、なぜ削除できないのか？
2. これは「あることを行う2つ目の方法」を導入していないか？
3. このコンポーネントは依然として単一のcontext窓に収まるか？
4. 型だけを読んで、LLMは正しくこれを使えるか？
5. `token/component` と `bundle/component` を悪化させていないか？

いずれかで詰まるなら、それはまだマージの準備ができていない。
