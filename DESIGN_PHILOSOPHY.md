# Axiom UI — 設計思想 / Design Philosophy

> An open-source UI library designed for the LLM as its primary consumer.
> LLMを第一の消費者として設計する、オープンソースUIライブラリ。

---

## 0. 一行のテーゼ

既存のUIライブラリは **人間のDX** に最適化されている。
Axiom UI は **LLMによる生成・理解・改変** に最初から最適化する。
人間はその副次的な受益者になる。

`shadcn/ui` が「結果的にAIフレンドリー」なら、Axiom は「設計思想からAIファースト」である。

---

## 1. 反転（The Inversion）

2026年、コードの主要な読み手はもはや人間だけではない。LLMが生成・デバッグ・リファクタの主体になった。
人間の認知負荷向けに積み上げられた抽象化（深い継承、多態、無数の設定オプション）は、
LLMにとっては **ハルシネーションを増やし、context窓とtokenを浪費する負債** になる。

Axiom はこの負債を持たない。設計上の全判断は、次の問いを通過する：

> 「この決定は、LLMがこのコンポーネントを **一度で正しく生成** する確率を上げるか、下げるか？」

上げないものは実装しない。

---

## 2. 設計原則 — マスクの「アルゴリズム」を規律にする

イーロン・マスクの5段階アルゴリズム（要求を疑う → 削除 → 簡素化 → 高速化 → 自動化）を、
そのままライブラリの設計規律として採用する。順序は不可逆である。

### 2.1 要求を疑う（Make requirements less dumb）
全てのprop、全てのコンポーネント、全ての設定に「なぜ必要か」を問う。
答えられない要求は、賢い人が作ったとしても愚かな要求である。まず要求を疑い、削れないか検討する。

### 2.2 削除する（The best part is no part）
- **最高のpropは「propがない」こと。** 規約で決まるものは設定にしない。
- **1つのことをやる方法は1つだけ（One way to do each thing）。** バリアントを増やさない。
  これはLLMのハルシネーションと生成の揺れを直接減らす、本ライブラリ最大の武器。
- **最高のコンポーネントは「コンポーネントがない」こと。** 素のHTML/ARIA primitiveで足りるものはラップしない。
- 削って10%も戻す必要が出ないなら、削り方が足りていない。

### 2.3 簡素化する（Simplify）
削除した後に残ったものだけを磨く。削る前に最適化しない。
残ったAPIには型と使用例を厚く与える。

### 2.4 高速化する（Accelerate — idiot indexを一級市民に）
「idiot index（本質コストに対する実コストの比）」をライブラリに適用する。
**コンポーネント1個あたりの token 数とバンドルサイズを計測し、CIでゲート化する。**
太ったコンポーネントはマージしない。これはそのまま「LLMが1ファイルを丸ごと安く速く読める」に効く。

### 2.5 自動化する（Automate — 最後に）
MCPサーバー、コード生成CLI、ドキュメント生成は、削って一本化した **後** に作る。
存在すべきでないもののための自動化は作らない。

## 3. アーキテクチャの帰結

原則から機械的に導かれる、具体的な構造上の決定。

### 3.1 1コンポーネント = 1ファイル、self-contained
実装・型・ドキュメント・使用例・機械可読manifestを同一ファイルに co-locate する。
LLMが1ファイルを ingest すれば、そのコンポーネントについて全てを知れる。
**「context窓に丸ごと入る」ことを設計制約とする。**

### 3.2 型が契約（Type as the contract）
`strict: true` 前提。状態は discriminated union で表現し、`any` を禁止する。
型定義そのものが一次ドキュメントとして機能し、LLMは型を読むだけで正しいコードを吐ける。

### 3.3 コピー&オーン方式（Vertical integration）
npmランタイム依存ではなく、CLIでユーザーのリポジトリにコードを注入する。
ユーザーがコードを所有する = ブラックボックスなし = LLMが自由に改変できる。

### 3.4 MCPファースト配布
ライブラリ本体と同時に MCPサーバー を提供する。
Claude Code / Cursor 等のエージェントが、バージョン整合の取れた仕様を token 無駄なく取得できる。
「docsを貼る」という作業自体を消す。

### 3.5 機械可読 manifest
各コンポーネントは `{ intent, props, states, a11y, examples }` を構造化データで持つ。
エージェントがコンポーネントを「説明」ではなく「選択して描画」できる（generative UI）土台にする。

### 3.6 ゼロコンフィグ・デフォルト
propなしで正しく動く。エスケープハッチは用意するが、必須にはしない。

## 4. 技術決定（v0）

思想（削除・一本化）に最も忠実な、オピニオネイテッドな1本の選択：

| 項目 | 決定 | 理由 |
|---|---|---|
| フレームワーク | **React** 専用 | 対象を絞り深く行く。抽象化レイヤを1つ削除する。 |
| スタイリング | **Tailwind CSS** | クラス文字列がそのままLLMに可読で、生成しやすい。 |
| 型 | **TypeScript strict** | 型 = 契約 = 一次ドキュメント。 |
| 配布 | **CLIによるコピー注入 + MCPサーバー** | ブラックボックス依存ゼロ。 |
| 依存 | **ランタイム依存を最小化** | idiot indexを下げる。 |

> この表は v0 の決定であり、思想の一部ではない。より良い理由があれば差し替える。
> ただし「選択肢を増やす」方向の変更は、原則2.2に反するため却下される。

---

## 5. Non-Goals（作らないもの）

明示的に削除した領域。これらは機能ではなく負債である。

- テーマの無限カスタマイズ機構（1つの良いデフォルトを持つ）
- 複数フレームワーク同時サポート（v0では）
- 「あらゆるユースケースに対応」を謳うことすべて
- 同じ目的を果たす2つ目のAPI
- 使われるか不明な機能のための設定オプション

---

## 6. 成功の測定基準

主観ではなく数値で判定する。

- **token/component**: 平均コンポーネントが単一のcontext窓に収まること。
- **bundle/component**: コンポーネント単位のバイト数を追跡し、退行させない。
- **one-shot generation rate**: 代表的なプロンプト集に対し、LLMが修正なしで正しいコードを生成する割合。これを主要KPIとする。
- **削除比率**: リリースごとに、追加した行数に対する削除した行数の比を記録する。

---

## 7. コントリビューション規律

全PRは、レビュー前に次を自問すること：

1. これは何かを **削除** しているか？ もしくは、なぜ削除できないのか？
2. これは「あることを行う2つ目の方法」を導入していないか？
3. このコンポーネントは依然として単一のcontext窓に収まるか？
4. 型だけを読んで、LLMは正しくこれを使えるか？
5. `token/component` と `bundle/component` を悪化させていないか？

いずれかで詰まるなら、それはまだマージの準備ができていない。

---

*The best part is no part. The best process is no process. — 本ライブラリは、これをUIに適用する試みである。*
